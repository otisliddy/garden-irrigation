# Irrigation System Wiring Diagram

## 1. Power Distribution

The charge controller has THREE terminal groups: PV (panel in), BAT (battery),
and LOAD (system output). The battery connects only to BAT. System loads are
powered from LOAD, not directly from the battery.

Charge controller battery mode: set to GEL (not "Sealed/Lithium" — the Gel
profile has the correct voltage thresholds and no equalisation pulses for LiFePO4).

L298N GND connects to the same Wago GND block as the buck converter.
Their VCC (12V motor power) is fed via the sleep-cutoff relay NO contact
(§9) — NOT directly from the +12V block.

NOTE: with "5V Enable jumper ON", each L298N's logic is powered from its
12V VCC through the onboard regulator. When the relay is de-energised the
whole module (logic + motor) is off — so firmware must keep all IN/EN pins
LOW whenever the relay is off, and only drive them after energising the
relay (see §10).


## 2. Signal / Data Side — authoritative pin map

ESP32-S3 DevKitC-1 on 44-pin screw shield.
Power: 5V from buck · GND bonded to battery GND at the Wago · 3V3 out to sensors.

```
GPIO  Dir  Function                         Connects to
────  ───  ───────────────────────────────  ──────────────────────────────────
  1   AIN  Soil sensor 1   (ADC1)           soil probe 1 AOUT  (cable 1, polytunnel)
  2   AIN  Soil sensor 2   (ADC1)           soil probe 2 AOUT  (cable 2, bed)
  3   AIN  Soil sensor 3   (ADC1)           soil probe 3 AOUT  (cable 2, bed)
  4   AIN  Soil sensor 4   (ADC1)           soil probe 4 AOUT  (cable 3, bed)
  5   AIN  Soil sensor 5   (ADC1)           soil probe 5 AOUT  (cable 3, bed)
  6   AIN  Battery voltage sense (ADC1)      divider tap (1MΩ/100kΩ, §9)
  8   IN   DHT11 DATA  (polytunnel)          DHT11 DATA  (module has onboard pull-up)
 10   OUT  Beds B  open    (L298N#1 IN1)     L298N #1 IN1
 11   OUT  Beds B  close   (L298N#1 IN2)     L298N #1 IN2
 12   OUT  Beds B  enable  (L298N#1 ENA)     L298N #1 ENA
 13   IN   Button — Beds B     (RTC / wake)  button → GND, 10k pull-up to 3V3
 14   IN   Button — Polytunnel (RTC / wake)  button → GND, 10k pull-up to 3V3
 16   OUT  Beds A  open    (L298N#1 IN3)     L298N #1 IN3
 17   OUT  Beds A  close   (L298N#1 IN4)     L298N #1 IN4   ⚠ UART1 TX default — do not start Serial1
 18   OUT  Beds A  enable  (L298N#1 ENB)     L298N #1 ENB   ⚠ UART1 RX default — do not start Serial1
 19   —    USB D-  (reserved)                native USB
 20   —    USB D+  (reserved)                native USB
 21   IN   Button — Beds A     (RTC / wake)  button → GND, 10k pull-up to 3V3
 39   OUT  Polytunnel open  (L298N#2 IN1)    L298N #2 IN1
 40   OUT  Polytunnel close (L298N#2 IN2)    L298N #2 IN2
 41   OUT  Polytunnel enable (L298N#2 ENA)   L298N #2 ENA
 42   OUT  Status LED                        220Ω → external LED → GND (§9)
 47   OUT  Relay control                     PN2222 base via 1k (§9)
```

WHY these pins (so the map is reproducible):
 - 5 soil + battery = six analog inputs on GPIO 1-6. ONLY ADC1 (GPIO 1-10)
   reads reliably while WiFi is on.
 - The DevKitC-1 left header is physically non-sequential. Each valve group
   is assigned to a physically adjacent triplet:
     Beds A    (16/17/18) → left  row positions  9/10/11 ✓
     Beds B    (10/11/12) → left  row positions 16/17/18 ✓
     Polytunnel (39/40/41)→ right row positions   9/8/7  ✓
   DHT11 (GPIO 8) sits at left row position 12 — a single wire, isolation fine.
 - Polytunnel is on the right row because the left row has no free adjacent
   triplet after Beds A and Beds B are placed. GPIO 39/40/41 are JTAG pins
   but usable as general-purpose digital outputs in production.
 - Relay moved to GPIO 47 to free GPIO 41 for the Polytunnel triplet.
 - Buttons on GPIO 13/14/21 — must be RTC-capable (GPIO 0-21) for EXT1
   deep-sleep wake. GPIO 0 is strapping/BOOT; not used for a button.
 - GPIO 17/18 are UART1 TX/RX defaults — safe as digital outputs as long as
   Serial1 is never started in firmware (it isn't by default).


## 3. Each L298N Channel → Solenoid Valve

```
   ESP32 GPIO ─► IN_x ┐                        ┌─► OUT_x ───┐
   ESP32 GPIO ─► IN_y ┤  L298N channel         │            │ 18 AWG
   ESP32 GPIO ─► EN_z ┘  (motor power 12V)     └─► OUT_y ───┤ outdoor
                                                            │ twin-core
                                                            ▼
                                                    Rain Bird 100-DV
                                                    9V latching solenoid

   Polarity doesn't matter for install — if "open" and "close" are
   reversed, swap them in firmware (or swap the wires).

   Drive sequence:
     Open  pulse: IN_x = HIGH, IN_y = LOW,  EN_z = HIGH for ~50ms,
                  then EN_z = LOW (or both INs LOW)
     Close pulse: IN_x = LOW,  IN_y = HIGH, EN_z = HIGH for ~50ms,
                  then EN_z = LOW
     Idle:        all LOW (zero current; valve holds last state)
```

                                  │  │     core 1 : DATA  (GPIO 8)
                                  │  │     core 2 : spare

                                  │  │     core 5 : soil 1 AOUT  (GPIO 1, bypasses perfboard)

## 8. ESP32-S3 GPIO Notes

- Most ESP32-S3 GPIO are flexible, but the §2 map is now fixed — the analog
  and RTC constraints below are why.
- Avoid: GPIO 19/20 (USB), 26-32 (flash), 33-37 (octal PSRAM on this N16R8),
  GPIO 0 / 45 / 46 (strapping), GPIO 38 (onboard RGB LED, DevKitC-1 v1.1) /
  GPIO 48 (onboard RGB LED, initial DevKitC-1 — check your board).
- GPIO 3 is also a strapping pin (JTAG source) but is fine as soil-sensor-3
  analog input — strapping only matters at boot, not for an ADC read.
- ADC1: GPIO 1-10 — the only ADC usable with WiFi on. Used for analog: GPIO
  1-6 (soil + battery). GPIO 8 and 10 are also ADC1-capable but used as
  digital I/O (DHT11 data, Beds B open); no conflict.
- ADC2: GPIO 11-20 → conflicts with WiFi; not used for any analog input here.
- RTC-capable (deep-sleep wake): GPIO 0-21 only. Buttons must live here.
- GPIO 17/18 are UART1 TX/RX defaults — safe as digital outputs; do not call
  Serial1.begin() without first reassigning Serial1 pins away from 17/18.
- GPIO 39/40/41 are JTAG pins (MTCK/MTDO/MTDI) but usable as digital outputs
  in production. GPIO 42 (MTMS) similarly fine.
- GPIO 47: relay control. GPIO 48 adjacent — do not confuse the two pins.


## 9. Prototype / Signal-Conditioning Board

A single perfboard inside the enclosure carries all the small signal
conditioning. Suggested zoning (left→right keeps the high-current relay
contacts away from the analog divider):

```
  [ DECOUPLING ]     [ BATTERY DIVIDER ]
   10µF + 104          1MΩ+100kΩ+104
   at power entry      → GPIO 6
  [ RELAY + DRIVER ]   [ BUTTON PULL-UPS ]  [ LED ]
   SONGLE relay +       10k × 3 to 3V3       220Ω
   PN2222 + 1k + 10k + 1N4007 (low-side)

  Decoupling sits at the far left, right where 3V3/GND enter from the ESP.
```

Four power rails span the board: +12V (from the Wago), and 3V3 / GND / +5V all
taken from ESP pins. There is NO separate 5V wire from the buck — the buck's
only output is the USB-C into the ESP, and the ESP's 5V pin re-exposes that USB
5V (it feeds the relay coil, the board's only 5V load). Each block taps the
rails it needs with a short stub (● = junction). See docs/perfboard_wiring.svg.

### 9d. Sleep-cutoff relay driver  (enhancement #2)

Use the SONGLE SRD-05VDC-SL-C — it's a BARE relay (2 coil pins + 3 contacts
NO/COM/NC, no onboard transistor/opto), so it solders flat onto the perfboard.
Drive its coil with one PN2222 as a low-side switch. This is NON-inverting
(GPIO HIGH = relay ON) and avoids the active-LOW 3.3V turn-off problem that
the TONGLING module has. Elegoo's tutorial drives this relay with an L293D,
but that quad half-bridge is overkill — a single transistor + flyback is the
standard minimal driver. (Keep the TONGLING module as a spare; see §9d-note.)

```
   +5V (ESP pin) ──┬────────────── coil A
                   │                    │
               [1N4007]             [coil]     SONGLE SRD-05VDC-SL-C
                   │                    │
                   └──────────── coil B ┘
                                        │  ← collector
   GPIO 47 ──[1k]──┬──────────── base ──┤  PN2222 (NPN)
                 [10k]                  │  ← emitter
                   │                    │
   GND ────────────┴────────────────────┘

   Relay CONTACTS (the 3 screw pins):
     COM ◄── +12V (Wago, post-fuse)        — use heavier wire for the
     NO  ──► L298N #1 VCC  +  L298N #2 VCC  — 12V load path on the board
     NC  ── not connected

   GPIO 47 HIGH → NPN saturates → coil energised → relay ON  (12V to L298Ns)
   GPIO 47 LOW/HiZ → NPN off    → coil off       → relay OFF (L298Ns isolated)

   - The 10k base→GND pull-down keeps the relay OFF at boot and through deep
     sleep (GPIO 47 is not latched) — no gpio_hold needed.
   - The bare relay has NO flyback diode (Elegoo relied on the L293D's
     internal clamps), so the 1N4007 across the coil is REQUIRED. If the board
     happens to have its own diode, the external one is harmless backup.
   - Coil ~71mA (5V/70Ω) only while energised — the brief awake window.

9d-note. TONGLING JQC-3FF-S-Z (the complete module: VCC/GND/IN + NO/COM/NC).
   It has an onboard driver but is ACTIVE-LOW and may not turn off reliably
   from a 3.3V HIGH, and it doesn't solder flat to perfboard. Kept as a spare.
   To mount a standalone module neatly: solder a 3-pin female header to the
   board for VCC/GND/IN and plug the module in via a short Dupont lead, or
   bolt it to the enclosure with M3 nylon standoffs and run a short ribbon.
```

## 10. Firmware Sequencing Rules (driven by the hardware above)

1. RELAY BEFORE VALVES. On wake: run initValve() (all IN/EN LOW) → set
   GPIO 47 HIGH → wait ~10ms for relay + L298N to settle → only then pulse a
   valve. Before sleep: all IN/EN LOW → GPIO 47 LOW → deep sleep. This stops
   the ESP back-powering an unpowered L298N's logic pins.

2. BOOT CLOSE-PULSE ONLY ON A REAL COLD BOOT. A 15-min deep-sleep timer wake
   re-runs setup(), but must NOT slam all valves closed (that would interrupt
   an intended watering). Check esp_reset_reason() / esp_sleep_get_wakeup_cause()
   and fire the close-all pulse only on a power-on reset, not on a timer/EXT1 wake.

3. BUTTON WAKE. Enable EXT1 wake (ESP_EXT1_WAKEUP_ANY_LOW) on the bitmask of
   GPIO 13/14/21. On wake, read esp_sleep_get_ext1_wakeup_status() to see which
   button fired, then toggle that zone's valve.

4. BATTERY READ. analogRead(GPIO 6) once per wake; scale by ×11 (divider).
   Alerting/health only — no safety action depends on it (fail-closed Layers
   3 & 4 are not implemented).

5. KNOWN TRADE-OFF (Layers 3 & 4 dropped). On a total power loss the ESP dies
   immediately; latching valves simply hold their last state until power
   returns and rule 2 fires the close-all. A valve left open before an outage
   stays open until then.
